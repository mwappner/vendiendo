Build a simple static web page to showcase second-hand items for sale.

The goal is a polished, friendly, modern one-page catalog, with no backend and no database server. All content should come from simple `.txt` files and image files stored in a local content folder.

The site will be hosted with GitHub Pages, so the final website must be fully static.

Project structure:

```txt
/
  index.html
  styles.css
  script.js
  build_manifest.py
  serve_local.py
  README.md
  content/
    landing.txt
    contact.txt
    products.json       <-- autogenerated by build_manifest.py
    products/
      coffee_table.txt
      coffee_table_1.png
      coffee_table_2.jpg
      lamp.txt
      lamp_1.webp
```

Content rules:

1. `content/landing.txt`

Contains the landing-page message. This text should appear in the first full-screen section of the website. Preserve line breaks reasonably.

2. `content/contact.txt`

Contains global contact information. This should appear in the catalog section, preferably near the top or bottom, and optionally repeated in each product card as a small contact reminder. Preserve line breaks reasonably.

3. Product files

Each product has one `.txt` file inside `content/products/`.

The filename stem defines the product ID.

Example:

```txt
coffee_table.txt
```

defines the product ID:

```txt
coffee_table
```

4. Product text format

Each product `.txt` has four lines:

```txt
Product name/description
Notes
Dimensions
Status
```

Example:

```txt
Solid wood coffee table
Small chip on one leg
20x30x10
SOLD
```

If a line is missing, format the card accordingly. For instance, note or dimension may not be available.

5. Product images

Images are stored in `content/products/`.

Images belonging to a product use the same filename stem as the product `.txt`. If more than one image for a product is available, the name is followed by `_1`, `_2`, `_3`, etc.

Example:

```txt
coffee_table.txt
coffee_table_1.png
coffee_table_2.jpg
coffee_table_3.webp
```

All of these should appear in the same product card.

Support these image formats:

```txt
.png
.jpg
.jpeg
.webp
```

Manifest generation:

Because browsers cannot automatically list local or hosted directory contents, include a Python script called:

```txt
build_manifest.py
```

This script should scan:

```txt
content/products/
```

and generate:

```txt
content/products.json
```

The manifest should be a rich manifest, not just a list of product IDs.

Example output:

```json
[
  {
    "id": "coffee_table",
    "text": "coffee_table.txt",
    "images": [
      "coffee_table_1.png",
      "coffee_table_2.jpg",
      "coffee_table_3.webp"
    ]
  },
  {
    "id": "lamp",
    "text": "lamp.txt",
    "images": [
      "lamp.webp"
    ]
  }
]
```

Rules for `build_manifest.py`:

* Use only the Python standard library.
* Scan `content/products/`.
* Detect every `.txt` file except hidden/system files.
* For each `.txt` file, use its stem as the product ID.
* Find matching images with the pattern:

```txt
product_id_1.ext
product_id_2.ext
product_id_3.ext
```

where `ext` is one of:

```txt
png, jpg, jpeg, webp
```

and the ending `_1` is optional.

* Write `content/products.json`.
* Use pretty JSON indentation.
* Print a short summary, for example:

```txt
Generated content/products.json with 8 products.
```

* If a product has no images, still include it with an empty `images` list.
* Make the script safe to run repeatedly.

Local testing:

Include a second Python script called:

```txt
serve_local.py
```

This script should:

* Optionally run `build_manifest.py` first, so the manifest is always fresh.
* Start a local static HTTP server.
* Use only the Python standard library.
* Serve the project root.
* Default to port `8000`.
* Print a helpful message like:

```txt
Serving at http://localhost:8000
Press Ctrl+C to stop.
```

This is needed because opening `index.html` directly by double-clicking may break JavaScript `fetch()` calls in the browser.

The intended local workflow should be:

```bash
python build_manifest.py
python serve_local.py
```

or, if `serve_local.py` automatically builds the manifest:

```bash
python serve_local.py
```

The intended GitHub Pages workflow should be:

```bash
python build_manifest.py
git add .
git commit -m "Update catalog"
git push
```

The committed repository should include the generated `content/products.json`, since GitHub Pages will only serve static files and will not run Python scripts.

Website behavior:

1. Landing section

* Full viewport height.
* Shows the text from `content/landing.txt`.
* Friendly modern layout.
* Include a clear “See items” button or scroll cue.
* Smooth scroll to the catalog section.

2. Scroll snapping

The page should have two main snap sections:

```txt
Landing section
Catalog section
```

Use CSS scroll snapping so scrolling from the landing section naturally lands on the catalog.

3. Catalog section

* Show the contact information from `content/contact.txt`.
* Show all product cards in a responsive grid.
* On desktop, use multiple columns.
* On mobile, use one column.
* Include a small toggle to hide/show sold items.

4. Product card design

Each card should contain:

* Image area at the top.
* Product name/description.
* Notes.
* Dimensions.
* Status badge.
* Optional contact reminder.

5. Image browsing

* If a product has one image, show it normally.
* If a product has multiple images, show them as a small carousel/gallery inside the card.
* Include previous/next buttons.
* Include small dot indicators or a “1 / 3” style counter.
* Images should be cropped nicely with `object-fit: cover`.
* Clicking an image should open it in a larger modal/lightbox view.
* Add keyboard navigation for the image modal if simple to implement.

6. Status styling

* If status is `VENDIDO`, show a clear sold badge and visually mute the card slightly.
* Other statuses should still be displayed as badges.
* Do not hide sold items by default, but include an easy hide/show sold toggle.

Style direction:

* Friendly, polished, modern, Canva-like.
* Soft background.
* Rounded cards.
* Subtle shadows.
* Large readable typography.
* Clean spacing.
* Pleasant hover effects.
* Fully responsive.
* No heavy framework required.
* Use plain HTML, CSS, and JavaScript.

Robustness:

* If a product has no images, show a nice placeholder instead of breaking.
* If a product `.txt` has fewer than four lines, fill missing fields with empty strings or sensible defaults.
* If `landing.txt`, `contact.txt`, or `products.json` fails to load, show fallback text.
* Keep the code simple and well-commented.
* Avoid unnecessary dependencies.

GitHub Pages deployment:

Add a section in `README.md` explaining how to deploy:

1. Create a GitHub repository.
2. Put all files in the repository.
3. Run:

```bash
python build_manifest.py
```

4. Commit and push:

```bash
git add .
git commit -m "Initial catalog site"
git push
```

5. In GitHub, go to:

```txt
Settings > Pages
```

6. Set the source to deploy from the main branch, root folder.
7. Use the generated GitHub Pages URL.

Deliverables:

* `index.html`
* `styles.css`
* `script.js`
* `build_manifest.py`
* `serve_local.py`
* `README.md`
* Example `content/` folder with:

  * `landing.txt`
  * `contact.txt`
  * autogenerated example `products.json`
  * at least two example products
  * example images or clearly marked placeholder image behavior

Code quality:

* Keep everything simple and readable.
* Use no framework unless absolutely necessary.
* Use semantic HTML where possible.
* Make the JavaScript modular enough that the manifest loading, text parsing, card rendering, carousel behavior, and modal behavior are understandable.
* Add helpful comments explaining how to add, remove, or edit products.
